## 6. Консилиум моделей (Model Ensemble) — Технология

### 6.1 Что такое «консилиум моделей»

Это подход, при котором несколько LLM работают над одной задачей параллельно, а затем их выводы агрегируются для получения результата лучшего качества, чем любая отдельная модель.

Базовые паттерны:

1. **Голосование (Voting):** N моделей → мажоритарный ответ
2. **Ранжирование (Ranking):** N моделей → scoring → лучший ответ
3. **Агрегация (Aggregation):** N моделей → синтез через агрегатор
4. **Многослойная агрегация (Layered MoA):** Proposers → Aggregator → (повтор)
5. **Дебаты (Debate):** M моделей дискутируют → консенсус
6. **Критика (Critique):** Генератор → N критиков → refinement

### 6.2 Together AI MoA (июнь 2024)

Оригинальная работа от Together AI:

- **Архитектура:** Layered MoA — несколько "proposer"-моделей генерируют ответы, затем "aggregator"-модель синтезирует финальный ответ
- **Результат:** 65.1% на AlpacaEval 2.0 (против 57.5% у GPT-4 Omni)
- **Статус:** открытый код (Apache 2.0), но не обновлялся с 2024
- **Минус:** жёсткая привязка к Together API, устаревшие модели в референсах

### 6.3 Современные решения (2025-2026)

**Вместо MoA от Together — актуальные подходы:**

#### 6.3.1 OpenRouter Model Routing
- Встроенный фолбэк между провайдерами
- Не ensemble в строгом смысле, но повышает надёжность

#### 6.3.2 OpenClaw Multi-Agent Orchestration
- Поддержка sub-agents (изолированные сессии)
- Можно запустить N параллельных sub-agent → собрать результаты → синтезировать
- **Это наш основной инструмент для консилиума**

#### 6.3.3 LangGraph / CrewAI
- Графовые оркестраторы для сложных multi-agent пайплайнов
- LangGraph: состояние, циклы, conditional routing
- CrewAI: role-based агенты с инструментами

#### 6.3.4 Anthropic's Model-in-the-Middle
- Claude как оркестратор: запускает подзадачи на других моделях через MCP/API
- Концепция «архитектор + исполнители»

#### 6.3.5 OmniRoute Multi-Provider Aggregation
- Сжатие промптов (RTK+Caveman) для экономии на дорогих моделях
- Роутинг между 231 провайдерами
- Можно настроить параллельные запросы к разным моделям

### 6.4 Ключевая научная работа (обновление)

Оригинальная работа, которую мы обсуждали: "Mixture-of-Agents" (arXiv 2406.04692, июнь 2024) — действительно устарела. Сравнивается с GPT-4 Omni.

**Что изменилось к июню 2026:**
- GPT-4 Omni давно снят с лидерства
- Появились reasoning-модели (DeepSeek R1/R4, Claude Opus thinking, o3/o4)
- MoA-подход эволюционировал в multi-agent orchestration
- Актуальный фокус — не просто агрегация, а *распределение ролей*: архитектор, кодер, ревьюер, тестировщик

## 7. Fusion Consult vs Консилиум

### 7.1 Что такое Fusion Consult (концепт)

Это предлагаемый навык для OpenClaw, который:
- Получает задачу
- Определяет какие модели нужны (роутинг)
- Запускает параллельные запросы к 2-4 моделям с разными «ролями»
- Агрегирует результаты в единый ответ

**Отличие от классического MoA:** не просто голосование/агрегация одинаковых промптов, а распределённая работа: одна модель — архитектор, другая — исполнитель, третья — критик.

### 7.2 Это та же технология?

**И да, и нет.**

| Аспект | Together MoA | Fusion Consult |
|--------|-------------|----------------|
| Параллельные запросы | Да | Да |
| Агрегация | Да (один aggregator) | Да (гибкая) |
| Ролевое разделение | Нет (все делают одно) | **Да — ключевое отличие** |
| Адаптивный роутинг | Нет | Да (выбор моделей под задачу) |
| Сложность | 1-3 слоя одинаковых запросов | DAG с разными промптами |
| Экономия | Нет (все N моделей = N× cost) | Частичная (разные модели для разных ролей) |

### 7.3 Что мы строим

**Fusion Consult = Консилиум с ролевым разделением**

Архитектура для кодинг-задачи:
```
Задача: "Напиши парсер CSV с валидацией"
│
├─ Архитектор (GLM-5.2 или Claude Opus): проектирует структуру
├─ Исполнитель (DeepSeek V4 Pro): пишет код
├─ Ревьюер (DeepSeek V4 Flash): проверяет на ошибки
└─ Синтезатор (DeepSeek V4 Pro): собирает финальный ответ
```

Архитектура для аналитической задачи:
```
Задача: "Проанализируй рыночные тренды X"
│
├─ Аналитик 1 (GLM-5.2): глубокий reasoning-анализ
├─ Аналитик 2 (Qwen3.7 Plus): альтернативная перспектива
├─ Факт-чекер (DeepSeek V4 Flash): верификация утверждений
└─ Синтезатор (DeepSeek V4 Pro или GLM-5.2): объединение
```

## 9. Claude Opus 4.8 — Анализ проблем

### 9.1 Что говорят пользователи

На основе данных от июня 2026:

- **Жалобы на ухудшение:** пользователи отмечают, что Anthropic «испортил» Opus в последних версиях
- **Over-thinking:** модель склонна к избыточному reasoning, тратит токены на размышления вместо действия
- **Стоимость:** $5.00/$25.00 через OpenRouter — при этом Opus 4.6 и 4.7 имеют ту же цену
- **Opus 4.8 Fast** стоит **вдвое дороже** ($10/$50) — позиционируется как «быстрый», но по факту просто дороже

### 9.2 Контекст деградации

Это часть более широкого тренда Anthropic в 2025-2026:
- Модели становятся «безопаснее» → менее полезные для реальных задач
- Чрезмерная цензура и отказы
- Claude Fable 5 (флагман) имеет AA Index 60, но стоит ~$7.70/M blended

### 9.3 Наша стратегия по Claude

- **Claude Opus 4.8 — НЕ автономный исполнитель**
- **Только как архитектор/оркестратор** для сверхсложных проектов
- **Один запрос → одна архитектурная схема** — не зацикливать
- **Никаких многочасовых сессий** — сожрёт весь бюджет
- **Использовать через OmniRoute** для снижения стоимости (сжатие промптов)

### Почему DeepSeek V4 Pro, а не GLM-5.2 как основной Pro?

1. **В 3 раза дешевле** при сопоставимом практическом качестве
2. **MoE-архитектура** лучше для кодинга (наша частая задача)
3. **Экосистема:** лучше интегрирован с OpenRouter, больше провайдеров
4. **GLM-5.2 используем точечно** — когда нужен максимальный reasoning

Zen оценивает сложность задачи и предлагает:
- **< 3 шагов:** Flash
- **3-5 шагов, код:** Pro
- **5+ шагов, анализ/стратегия:** GLM-5.2
- **Критическая архитектура:** Claude Opus 4.8 (с подтверждением Дмитрия)
